## 1. Design Goals

### 1.1 Core Principle

```
Chitu Intelligence = Intelligence Layer (discover, research, score, draft)
EspoCRM = Execution Layer (manage, follow-up, sell, close)
```

Neither system duplicates the other's responsibility. The boundary is:
- **Chitu owns everything before a human decides to act.**
- **EspoCRM owns everything after a human decides to act.**

### 1.2 Design Constraints

| Constraint | Rule |
|---|---|
| One-way data flow | Chitu → EspoCRM only. CRM never writes back to Engine. |
| Human-gated conversions | Lead→Account, Lead→Opportunity require explicit CRM user action. |
| Immutable provenance | Engine-scored fields are snapshots; CRM cannot overwrite them. |
| No auto-send | Email drafts are generated by Chitu but never sent without human approval. |
| Idempotent sync | Same Engine payload → same CRM result; no duplicate Leads. |

### 1.3 What This Design Replaces

The legacy `leads.lead_status` state machine (13 ad-hoc values, no validation, scattered across 6+ files) is **fully replaced** by this design. The Prospecting Engine state machine (`prospecting_engine/domain/state_machine.py`) governs the Engine side; this design governs the CRM side and the handshake between them.

### 2.2 Lifecycle States — Detailed Definitions

#### Engine-Side States (Chitu Intelligence)

These are defined in `prospecting_engine/domain/state_machine.py` and are authoritative for the Engine. They are restated here for CRM workflow context.

| # | State | Owner | Meaning |
|---|-------|-------|---------|
| 1 | `DISCOVERED` | Engine | Raw candidate from search/import. No research yet. |
| 2 | `PRE_SCREENED` | Engine | ICP filter, dedup, official-brand exclusion applied. |
| 3 | `RESEARCH_QUEUED` | Engine | Queued for website research. |
| 4 | `RESEARCHING` | Engine | Website research in progress. |
| 5 | `RESEARCHED` | Engine | Website research complete; evidence collected. |
| 6 | `SCORE_QUEUED` | Engine | Queued for Canonical V4 scoring. |
| 7 | `SCORING` | Engine | Scoring in progress. |
| 8 | `SCORED` | Engine | Score computed; tier assigned. |
| 9 | `ENRICHMENT_QUEUED` | Engine | Queued for contact enrichment. |
| 10 | `ENRICHING` | Engine | Contact enrichment in progress. |
| 11 | `ENRICHED` | Engine | Contacts collected; all gates satisfied. |
| 12 | `OUTREACH_READY` | Engine | **Sync gate:** eligible for CRM import. |
| 13 | `APPROVED_FOR_CRM` | Engine | Human (or rule) flagged for sync. |
| 14 | `SYNCED_TO_ESPO` | Engine | Payload delivered to CRM; idempotency key stored. |
| 15 | `REJECTED_BUSINESS` | Engine | Terminal: ICP/brand/platform/media exclusion. |
| 16 | `FAILED_TECHNICAL` | Engine | Website down, timeout, CAPTCHA. Retry-eligible. |
| 17 | `RETRY_PENDING` | Engine | Waiting for retry after technical failure. |

#### CRM-Side States (EspoCRM)

These are EspoCRM Lead statuses managed by CRM workflows and human actions. These replace the legacy `leads.lead_status` free-text values.

| # | CRM Lead Status | Trigger | Meaning |
|---|-----------------|---------|---------|
| 1 | `New` | Engine sync creates Lead | Freshly imported; awaiting CRM review. |
| 2 | `Research Reviewed` | CRM user reviews evidence | Human has verified research quality. |
| 3 | `Qualified` | CRM user confirms score/fit | Lead is worth pursuing. Eligible for outreach. |
| 4 | `Unqualified` | CRM user rejects | Not a fit. Terminal for this round. Engine may re-discover later. |
| 5 | `Outreach Ready` | CRM user approves draft | Email draft reviewed and approved. Ready for human send. |
| 6 | `Contacted` | Human sends email (manual) | First outreach sent. Waiting for reply. |
| 7 | `Engaged` | Reply received (manual update or webhook) | Dealer replied. Requires follow-up. |
| 8 | `Meeting Scheduled` | CRM user creates Meeting | Meeting booked with dealer. |
| 9 | `Opportunity Created` | CRM user converts to Opportunity | Sales pursuit formally opened. |
| 10 | `Converted` | Lead → Account + Contact + Opportunity | Full conversion. Lead record closed. |
| 11 | `Nurturing` | CRM user defers | Not now, but keep. Periodic review. |
| 12 | `Invalid` | CRM user marks bad data | Wrong company, dead domain, etc. |

### 2.3 State Transition Rules (CRM Side)

```
New ──────────→ Research Reviewed ──────────→ Qualified ──────────→ Outreach Ready
 │                    │                            │                      │
 │                    │                            │                      │
 └──→ Invalid         └──→ Unqualified             └──→ Nurturing         │
                                                                         │
 Outreach Ready ──────────→ Contacted ──────────→ Engaged                │
       │                        │                    │                    │
       │                        │                    ├──→ Meeting Scheduled
       │                        │                    │                    │
       └──→ (back to Qualified) │                    └──→ Opportunity Created
                                │                                         │
                                └──→ Nurturing (no reply after sequence)  │
                                                                          │
 Engaged ──────────→ Opportunity Created ──────────→ Closed Won ──→ Customer
       │                        │                          │
       │                        │                     Closed Lost
       └──→ Nurturing           │
                                │
 Converted ──→ Lead closed; lifecycle continues on Account + Opportunity
```

**Rules:**
- `New` → `Invalid` or `Unqualified`: immediate, no dependencies.
- `New` → `Research Reviewed`: requires at least one CRM user opening the evidence detail view.
- `Research Reviewed` → `Qualified`: requires CRM user confirming score/fit (checkbox or status change).
- `Qualified` → `Outreach Ready`: requires email draft existence AND human approval.
- `Outreach Ready` → `Contacted`: requires email actually sent (manual action; not automated).
- `Contacted` → `Engaged`: requires reply received (manual flag or email webhook).
- `Engaged` → `Opportunity Created`: requires Opportunity record linked to Lead.
- `Opportunity Created` → `Converted`: requires Account + Contact + Opportunity all linked.
- Any active state → `Nurturing`: CRM user defers; set follow-up reminder.
- `Nurturing` → `Qualified`: CRM user re-activates.

### 3.2 Lead Object

**Purpose:** Intelligence import target. Represents a prospecting candidate before human qualification.

| Field Group | Fields | Source | Mutable by CRM? |
|---|---|---|---|
| **Identity** | `name`, `website`, `country` | Engine import | CRM may correct `name` only |
| **Engine Identity** | `peCandidateId`, `peCanonicalDomain` | Engine import | **No** (immutable) |
| **Score Snapshot** | `peOpportunityScoreV4`, `peScoreTier`, `peConfidence`, `peEvidenceCoverage` | Engine import | **No** (snapshot) |
| **Recommendation** | `peBestFirstProduct`, `peCrossSellPath` | Engine import | **No** (snapshot) |
| **Qualification** | `peQualificationStatus`, `peResearchStatus`, `peCustomerType` | Engine import | **No** (snapshot) |
| **Provenance** | `peEngineVersion`, `peScoreRulesVersion`, `peEvidenceSchemaVersion`, `peRegistryVersion`, `peEvidenceSnapshotHash` | Engine import | **No** (immutable) |
| **Sync Metadata** | `peSyncStatus`, `peSyncedAt` | Extension-owned | **No** (extension-managed) |
| **CRM Lifecycle** | `status` (Lead status enum), `description` | CRM user | **Yes** |
| **CRM Ownership** | `assignedUser`, `teams` | CRM user | **Yes** |
| **CRM Source** | `leadSource` (mapped from Engine `source.channel`) | Engine import | CRM may refine |
| **Research Evidence** | `researchEvidences` (link to ResearchEvidence) | Engine import | **No** (immutable, append-only) |
| **Contacts** | `contacts` (link to Contact records discovered during enrichment) | Engine import | CRM may add |

**Lead Status Field:** EspoCRM enum field `status` replaces legacy `leads.lead_status`. Values defined in §2.2 CRM-Side States.

### 3.3 Account Object

**Purpose:** Represents a confirmed company relationship. **Created only by CRM user action**, never by Engine import.

| Field | Source | Notes |
|---|---|---|
| `name` | Copied from `Lead.name` | CRM user may edit |
| `website` | Copied from `Lead.website` | CRM user may edit |
| `billingAddressCountry` | Copied from `Lead.country` | Nullable |
| `type` | CRM user selects | Customer, Partner, Reseller, etc. |
| `peCanonicalDomain` | Copied from `Lead.peCanonicalDomain` | Retained for dedup |
| `peEngineSnapshotHash` | Copied from `Lead.peEvidenceSnapshotHash` | Provenance link |
| `assignedUser`, `teams` | CRM user assigns | — |
| `description` | CRM user writes | Free-text CRM notes |

**Creation rules:**
- Only from a Lead in `Qualified` status or higher.
- CRM user explicitly clicks "Convert to Account."
- System pre-fills from Lead; user confirms/edits.
- Deduplication check by `peCanonicalDomain` and website before creation.

### 3.4 Contact Object

**Purpose:** A person at a dealer company. **Dual-source:** Engine enrichment AND CRM user entry.

| Field | Source | Notes |
|---|---|---|
| `firstName`, `lastName` | Engine enrichment or CRM user | Parsed from email signature if available |
| `emailAddress` | Engine enrichment | Primary contact email |
| `phoneNumber` | Engine enrichment | If found |
| `title`, `department` | CRM user | — |
| `account` | CRM user links | After Account conversion |
| `lead` | Engine sync links | During Lead phase |
| `source` | `"Engine Enrichment"` or `"CRM Manual"` | — |
| `peContactChannel` | Engine enrichment | sales, info, support, contact, etc. |

**Creation rules:**
- Engine sync may create Contact records linked to Lead during enrichment import.
- Contacts imported by Engine are read-only (`source = "Engine Enrichment"`); CRM user may copy and edit a duplicate for CRM-managed contacts.
- CRM user may create new Contacts manually at any Lead stage.
- On Lead→Account conversion, Contacts may be re-linked to Account.

### 3.5 Opportunity Object

**Purpose:** A qualified sales pursuit. **Created only by CRM user**, never by Engine import.

| Field | Source | Notes |
|---|---|---|
| `name` | CRM user | Suggested: `{Company} — {Product}` |
| `account` | Linked Account | After conversion |
| `lead` | Source Lead | Before Account conversion |
| `amount` | CRM user | Not estimated by Engine |
| `probability` | CRM user | Follows sales stage defaults |
| `stage` | CRM user | From EspoCRM Sales Pipeline |
| `closeDate` | CRM user | — |
| `productInterest` | Copied from `Lead.peBestFirstProduct` | Advisory only |
| `scoreTier` | Copied from `Lead.peScoreTier` | Advisory only |
| `assignedUser`, `teams` | CRM user | — |

**EspoCRM Sales Pipeline Stages (default proposal):**

| Stage | Probability | Meaning |
|---|---|---|
| `Prospecting` | 10% | Initial outreach or response received |
| `Qualification` | 25% | Need confirmed; budget discussed |
| `Proposal` | 50% | Quote or proposal sent |
| `Negotiation` | 75% | Terms being discussed |
| `Closed Won` | 100% | Deal signed |
| `Closed Lost` | 0% | Deal lost |

| Field | Notes |

**Purpose:** Outgoing and incoming email records. **Outgoing created by CRM user; incoming from email integration.**

| Field | Notes |
|---|---|
| `status` | Draft, Sent, Archived, Received |
| `from`, `to`, `cc`, `bcc` | Addresses |

**Important:** Engine generates email drafts as **EspoCRM Email records with status=Draft**. These are never auto-sent. CRM user reviews, edits if needed, and manually sends.

## 4. Automation Rules

### 4.1 Design Principle

Automation rules are split into two tiers:

| Tier | Location | Authority | Examples |
|---|---|---|---|
| **Sync-time rules** | Engine → CRM import logic | Chitu Intelligence | Gate, dedup, field mapping |
| **CRM Workflow rules** | EspoCRM Formula/Sandbox or Workflow engine | EspoCRM | Task creation, status changes, notifications |

**Rule:** CRM workflows never call back to the Engine. Engine import never triggers emails or creates Opportunities.

### 4.2 Sync-Time Rules (Chitu Intelligence Side)

These execute during `POST /api/v1/prospecting-engine/import` and are defined in `integration/espocrm_sync/gate.py` and `contract.py`.

| # | Rule | Condition | Action |
|---|------|-----------|--------|
| S1 | **Sync Gate** | `qualification.status == "OUTREACH_READY"`, score tier A/B/C, coverage ≥ 0.50, confidence ≥ 0.60 | Allow import |
| S2 | **Version Gate** | `score.rules_version == "canonical-scoring-v4.0"`, contract version recognized | Allow import; reject unknown versions |
| S3 | **Exclusion Gate** | `provenance.official_brand_excluded == false`, no business exclusion | Allow import |
| S4 | **Dedup by Domain** | `peCanonicalDomain` matches existing Lead | Update Engine-owned fields (idempotent); return existing Lead |
| S5 | **Dedup by Idempotency Key** | Same `idempotency_key` + same `payload_hash` | Return `ALREADY_SYNCED`; no mutation |
| S6 | **Contact Import** | Enrichment data present in payload | Create Contact records linked to Lead |
| S7 | **Evidence Import** | Evidence items present in payload | Create/update ResearchEvidence records linked to Lead |
| S8 | **Email Draft Import** | Email drafts present in payload (future) | Create EspoCRM Email records with status=Draft |

### 4.3 CRM Workflow Rules (EspoCRM Side)

These execute as EspoCRM Workflow or Formula rules after Lead creation/update. **None of these call the Engine.**

#### 4.3.1 Lead Status Automation

| # | Trigger | Condition | Action |
|---|---------|-----------|--------|
| W1 | Lead created (sync) | `peScoreTier == "A"` AND `peOpportunityScoreV4 >= 80` | Create Task: "Review High-Score Lead: {Lead.name}" assigned to CRM admin; due in 24 hours; priority High |
| W2 | Lead created (sync) | `peScoreTier == "B"` | Create Task: "Review Lead: {Lead.name}" assigned to CRM admin; due in 72 hours; priority Normal |
| W3 | Lead created (sync) | `peScoreTier == "C"` | No auto-task; Lead appears in "New Leads" queue for manual triage |
| W4 | Lead status → `Contacted` | `status == "Contacted"` | Create Task: "Follow-up: {Lead.name}" due in 5 days (after Email 1); set `peOutreachSequence = 1` |
| W5 | Lead status → `Engaged` | `status == "Engaged"` | Create Task: "Reply to {Lead.name}" due in 24 hours; priority High; if Opportunity doesn't exist, create Task: "Qualify {Lead.name} for Opportunity" |
| W6 | Lead status → `Contacted` + 5 days elapsed | No reply; `peOutreachSequence == 1` | Create Task: "Send Email 2 to {Lead.name}" (secondary product); set `peOutreachSequence = 2` |
| W7 | Lead status → `Contacted` + 12 days elapsed | No reply; `peOutreachSequence == 2` | Create Task: "Send Email 3 to {Lead.name}" (third product, soft close); set `peOutreachSequence = 3` |
| W8 | Lead status → `Contacted` + 21 days elapsed | No reply after full 3-email sequence | Auto-transition to `Nurturing`; create Task: "Re-evaluate {Lead.name}" due in 90 days |
| W9 | Meeting status → `Held` | Meeting linked to Lead | Create Task: "Meeting follow-up: {Lead.name}" due in 24 hours |

#### 4.3.2 Opportunity Automation

| # | Trigger | Condition | Action |
|---|---------|-----------|--------|
| W10 | Opportunity created | Linked to Lead | Copy `Lead.peBestFirstProduct` → `Opportunity.productInterest`; copy `Lead.scoreTier` → `Opportunity.scoreTier` |
| W11 | Opportunity stage → `Proposal` | — | Create Task: "Prepare proposal for {Opportunity.name}" due in 48 hours |
| W12 | Opportunity stage → `Closed Won` | — | Create Task: "Onboard {Account.name} — send welcome package" due in 24 hours |
| W13 | Opportunity stage → `Closed Lost` | — | Create Task: "Lost deal analysis: {Opportunity.name}" due in 7 days |

#### 4.3.3 Account Automation

| # | Trigger | Condition | Action |
|---|---------|-----------|--------|
| W14 | Account created | Conversion from Lead | Link all ResearchEvidence from Lead to Account; link all Contacts from Lead to Account |
| W15 | Account created | — | Create Task: "Complete Account profile for {Account.name}" due in 48 hours |

#### 4.3.4 Email Automation (Future — when email integration is live)

| # | Trigger | Condition | Action |
|---|---------|-----------|--------|
| W16 | Email received | Matches tracked Lead/Contact email address | Auto-link to Lead; if Lead status is `Contacted`, update to `Engaged` |
| W17 | Email bounced | Matches tracked Lead | Flag Lead; create Task: "Bounced email for {Lead.name} — verify contact" |
| W18 | Email auto-responder | "Out of office" or vacation reply | Log on Lead timeline; do not change status; adjust follow-up timing |

### 4.4 Explicit Non-Automations

These actions must **never** be automated. They require explicit human action:

| Action | Why Not Automated |
|---|---|
| Sending email | EMAIL_WRITING_RULES_V3: human sends only |
| Creating Opportunity | Requires human judgment on sales pursuit viability |
| Creating Account | Requires human confirmation of company relationship |
| Changing `pe*` fields | Engine provenance fields are immutable |
| Deleting ResearchEvidence | Audit trail must be preserved |
| Bulk-converting Leads | Each Lead requires individual human review |
| Auto-responding to dealer replies | Replies require human judgment |

| `DELETE` | `/api/v1/ResearchEvidence/{id}` | 3A-2.x | Synthetic test rollback only |
| `DELETE` | `/api/v1/Lead/{id}` | 3A-2.x | Synthetic test rollback only |

#### 5.2.2 CRM Internal (EspoCRM Workflow/Formula)

| Trigger | Scope | Phase |
|---|---|---|
| Lead created | Create tasks based on score tier | 3A-25 |
| Lead status changed | Cascade task creation | 3A-25 |
| Opportunity stage changed | Create follow-up tasks | 3A-25 |
| Meeting scheduled/held | Create follow-up tasks | 3A-25 |
| Email received (webhook) | Link to Lead; update status | 3A-27 |

#### 5.2.3 What the Engine NEVER Calls

| Forbidden Endpoint | Reason |
|---|---|
| `POST /api/v1/Account` | Account creation is human-gated |
| `POST /api/v1/Opportunity` | Opportunity creation is human-gated |
| `PUT /api/v1/Lead/{id}/status` | CRM lifecycle is CRM-owned |
| `PUT /api/v1/Lead/{id}/assignedUser` | CRM ownership is CRM-owned |
| `POST /api/v1/Email` (send) | Email sending is human-only |
| `DELETE /api/v1/Lead/{id}` (production) | Deletion is CRM admin action |
| Any bulk-update endpoint | Each record requires individual treatment |

## 6. Workflow Stages in Detail

### 6.1 Stage: Lead Import (Engine → CRM)

```
TRIGGER: Engine state → SYNCED_TO_ESPO
ACTOR: Chitu Intelligence (sync adapter)

1. Build SyncSource from Engine models
2. Evaluate sync gate (score, tier, coverage, confidence, exclusions)
3. Map to Sync Contract V1 payload
4. POST to EspoCRM /api/v1/prospecting-engine/import
5. EspoCRM validates schema + gate (dual validation)
6. EspoCRM resolves idempotency (same key → return existing)
7. EspoCRM creates/updates Lead (Engine-owned fields only)
8. EspoCRM creates/updates ResearchEvidence records
9. EspoCRM creates/updates Contact records (if enrichment data)
10. EspoCRM links: Lead → ResearchEvidence, Lead → Contact
11. EspoCRM sets peSyncStatus = "SYNCED", peSyncedAt = now
12. Return receiver result to Engine

CRM WORKFLOW FIRES:
  - W1/W2/W3: Score-tier-based task creation (see §4.3.1)

HUMAN ACTION REQUIRED:
  - Review Lead in CRM
  - Verify research quality
  - Qualify or Reject
```

### 6.2 Stage: Lead Qualification (CRM)

```
TRIGGER: CRM user opens Lead detail view
ACTOR: CRM user

1. Review Lead summary: score, tier, best_first_product, evidence count
2. Review ResearchEvidence items (linked, read-only)
3. Review Contacts (if enrichment found emails/phones)
4. Decision:
   a. Approve → status = "Qualified"
   b. Reject → status = "Unqualified" (with reason note)
   c. Defer → status = "Nurturing" (with follow-up date)

CRM WORKFLOW FIRES:
  - (Qualified): Lead moves to Outreach Ready queue
  - (Unqualified): No further action; Engine may re-discover
  - (Nurturing): Task created for follow-up review

HUMAN ACTION REQUIRED:
  - Decision must be explicit (click, not automatic)
```

### 6.3 Stage: Email Preparation (CRM)

```
TRIGGER: Lead status = "Qualified"
ACTOR: Chitu Intelligence (future email draft import) + CRM user

1. (Future) Engine imports email drafts as EspoCRM Email records (status=Draft)
2. CRM user opens Lead → Email drafts
3. CRM user reviews draft:
   - Primary product correct? (Best_First_Product)
   - Contact email correct? (from enrichment)
   - Personalization accurate?
   - No forbidden language? (per EMAIL_WRITING_RULES_V3)
4. CRM user edits if needed
5. CRM user changes Lead status → "Outreach Ready"

CRM WORKFLOW FIRES:
  - None at this stage (waiting for human send)

HUMAN ACTION REQUIRED:
  - Draft review and approval
  - Draft editing (if needed)
  - Status change to "Outreach Ready"
```

### 6.4 Stage: First Contact (CRM)

```
TRIGGER: CRM user sends email
ACTOR: CRM user

1. CRM user opens approved Email draft
2. CRM user clicks "Send" (manual action — NEVER automated)
3. EspoCRM sends email via configured SMTP/email integration
4. Email record status → "Sent"
5. Lead status → "Contacted"
6. peOutreachSequence = 1 (first email)

CRM WORKFLOW FIRES:
  - W4: Create "Follow-up in 5 days" task
  - W6: (after 5 days, if no reply) Create "Send Email 2" task

HUMAN ACTION REQUIRED:
  - Manual send click
  - (Future) Manual sequence management
```

### 6.5 Stage: Engagement (CRM)

```
TRIGGER: Dealer replies to email
ACTOR: EspoCRM email integration (webhook) or CRM user (manual)

1. Email received → matched to Lead by email address
2. Email linked to Lead timeline
3. Lead status → "Engaged"

CRM WORKFLOW FIRES:
  - W5: Create high-priority "Reply to dealer" task
  - W5: If no Opportunity exists, create "Qualify for Opportunity" task

HUMAN ACTION REQUIRED:
  - Read and respond to dealer's email
  - Assess interest level
  - Decide: create Opportunity, schedule Meeting, or continue email discussion
```

### 6.6 Stage: Opportunity Management (CRM)

```
TRIGGER: CRM user creates Opportunity from Lead
ACTOR: CRM user

1. CRM user clicks "Create Opportunity" from Lead
2. EspoCRM pre-fills:
   - name: "{Lead.name} — {Lead.peBestFirstProduct}"
   - productInterest: Lead.peBestFirstProduct
   - scoreTier: Lead.peScoreTier
   - lead: linked to source Lead
3. CRM user fills:
   - amount (estimated deal value)
   - closeDate (expected close)
   - stage: "Prospecting" (default)
   - assignedUser
4. Opportunity created; Lead status → "Opportunity Created"

CRM WORKFLOW FIRES:
  - W10: Copy product/scoring to Opportunity
  - W11-W13: Stage-based task creation (future stages)

HUMAN ACTION REQUIRED:
  - All Opportunity fields set by human
  - Stage progression managed by human
  - Close Won/Lost decision by human
```

### 6.7 Stage: Account Conversion (CRM)

```
TRIGGER: CRM user converts Lead to Account (typically after Opportunity Closed Won)
ACTOR: CRM user

1. CRM user clicks "Convert to Account" from Lead
2. EspoCRM pre-fills from Lead
3. Dedup check by peCanonicalDomain
4. CRM user confirms/edits Account fields
5. Account created
6. Contacts re-linked from Lead to Account
7. ResearchEvidence re-linked from Lead to Account
8. Lead status → "Converted" (Lead record closed)

CRM WORKFLOW FIRES:
  - W14: Link evidence + contacts to Account
  - W15: Task to complete Account profile

HUMAN ACTION REQUIRED:
  - Conversion decision
  - Account field review
  - Contact re-linking confirmation
```

### 6.8 Stage: Nurturing & Reactivation (CRM)

```
TRIGGER: CRM user defers Lead; periodic review timer
ACTOR: CRM user + CRM Workflow

1. CRM user sets Lead status → "Nurturing"
2. CRM user sets follow-up date
3. CRM Workflow creates Task for follow-up date

TWO REACTIVATION PATHS:

Path A — CRM user reactivates:
  1. Task fires at follow-up date
  2. CRM user reviews Lead
  3. Decision: return to "Qualified" or stay "Nurturing"

Path B — Engine re-discovers:
  1. Engine periodic re-search finds same domain
  2. Engine produces new score/evidence
  3. New sync payload generated
  4. EspoCRM idempotency check: same peCanonicalDomain
  5. Idempotent update: Engine-owned fields updated; CRM fields preserved
  6. If Lead status was "Unqualified" or "Nurturing", CRM user notified of new intelligence
  7. CRM user decides: re-qualify or ignore

HUMAN ACTION REQUIRED:
  - All reactivation decisions
```

## 8. Risk Register

| # | Risk | Likelihood | Impact | Mitigation |
|---|------|-----------|--------|------------|
| R1 | EspoCRM Workflow engine limitations prevent complex rules | Medium | Medium | Start with simple rules; escalate complex logic to custom Formula scripts |

| R6 | Contact data from enrichment is stale/wrong | Medium | Medium | Contacts imported as read-only; CRM user can override with manual entries |
| R7 | EspoCRM upgrade breaks custom entity/field definitions | Low | High | Extension uses supported metadata conventions; test after each EspoCRM upgrade |
| R8 | Engine sync volume exceeds EspoCRM API rate limits | Low | Low | Batch sync with idempotency; small dealer volumes expected (hundreds, not millions) |

## 9. Open Decisions

These items require explicit stakeholder decision before Phase 3A25 implementation begins:

| # | Decision | Options | Recommendation |
|---|----------|---------|----------------|
| D1 | Should Contact records from Engine enrichment be editable by CRM users? | A) Read-only (CRM creates copies for edits) B) Editable (CRM can modify Engine imports) | **A — Read-only.** Preserves enrichment provenance. CRM user creates manual Contact for their managed version. |
| D2 | Should Engine email draft import create Draft Email records, or should Engine keep drafts and CRM only reference them? | A) Import as EspoCRM Email Drafts B) Keep in Engine; CRM has "View Draft" link | **A — Import as Drafts.** Single source of truth in CRM; CRM user edits and sends from one place. |

Override with known names. |
| D4 | Should "Contacted → Engaged" status change be automatic (via email webhook) or manual (CRM user sets after seeing reply)? | A) Automatic via webhook B) Manual | **Both.** Webhook auto-updates as primary path; manual override available as fallback. |
